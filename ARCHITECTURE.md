# خطة المعمارية والتنفيذ الشاملة لمشروع Anime Translator (GUI + Core)

## 0) الهدف النهائي
بناء نظام **Desktop App (GUI)** لترجمة ملفات ترجمة الأنمي (SRT/ASS) بالاعتماد على نماذج اللغة الكبيرة (LLM).
**الميزات الأساسية المدعومة:**
* إدارة مشاريع مبنية على اسم الأنمي (Anime-based).
* ترجمة مستقرة وذات سياق.
* دعم قواميس (Glossaries) وذاكرة مصطلحات (Term Memory) متقدمة.
* إمكانية التعديل اليدوي للبيانات والقاموس عبر واجهة المستخدم (GUI).
* تحمل فشل الاتصال بالشبكة (Fault Tolerance) والاستئناف التلقائي.

---

## 1) المعمارية العامة (Architecture)
يُبنى النظام على شكل طبقات منفصلة لضمان استقرار الواجهة وسهولة الصيانة:

```text
[ GUI (PySide6) ] 
       ↓ 
[ Config Layer ]
       ↓ 
[ Runner (Subprocess) ]
       ↓ 
[ pipeline.py (Core Engine) ]
       ↓ 
[ State + Logs + Project Data ]
```

---

## 2) مفهوم النظام (مهم جدًا)
يعتمد النظام على هيكلة المشاريع بناءً على اسم العمل:
* **Project (Anime):** يمثل مسلسل الأنمي بأكمله.
  * `data/`: مجلد مشترك لجميع حلقات هذا الأنمي.
  * `episodes/`: المجلدات الخاصة بكل حلقة.

**القاعدة الذهبية:**
* **البيانات (Data):** (القاموس، الشخصيات، سياق العمل، ذاكرة المصطلحات) تكون على **مستوى المشروع (Project-level)** وتُشارك بين كل الحلقات.
* **الترجمة (Translation):** تكون على **مستوى الحلقة (Episode-level)**.

---

## 3) هيكل الملفات النهائي
```text
projects/
  attack-on-titan/
    data/
      glossary.json
      characters.json
      work_context.json
      term_memory.json
    episodes/
      s01e01/
        raw.srt
        chunks/
        logs/
        output.srt
        state.json

config/
  user_settings.json

gui/
  main.py
  views/
  services/
```

---

## 4) التدفق الداخلي للبيانات (Pipeline)
1. **Input file:** استلام ملف الترجمة.
2. **Project Resolution:** تحديد الأنمي والحلقة.
3. **Bootstrap:** بناء المشروع ومجلداته إذا كانت هذه المرة الأولى.
4. **Parse (SRT/ASS):** تحليل النص واستخراجه.
5. **Normalize:** تنظيف النصوص.
6. **Chunking:** تقسيم النص لشنكات.
7. **Execution Loop (لكل الشنكات):**
   * Build Context (بناء السياق المسبق واللاحق).
   * LLM Call (الاستدعاء).
   * JSON Extractor (استخراج الترجمة من الرد).
   * Validator (التدقيق البرمجي).
   * Constraint Engine (تطبيق القيود CPS، التقسيم، الضغط).
8. **Save:** حفظ حالة الشنك.
9. **Update Term Memory:** تحديث الذاكرة.
10. **Rebuild file:** تجميع الملف النهائي.

---

## 5) تحليل المشروع (Project Resolution)
* **المدخل:** اسم الملف (مثال: `Attack.on.Titan.S01E01.srt`).
* **الخطوات:** 
  * استخراج اسم الأنمي (Anime) ورقم الحلقة (Episode).
  * توحيد التسمية (Normalize): تحويل `Attack on Titan` إلى `attack-on-titan`.
  * التحقق من وجود المشروع.. إذا لم يكن موجوداً → **Bootstrap**.

---

## 6) التأسيس (Bootstrap - إنشاء المشروع)
يقوم بإنشاء المجلدات الأساسية لـ `projects/{anime}/data/` والملفات التالية بشكل فارغ أو افتراضي:
* `glossary.json` → `{ "terms": [] }`
* `characters.json` → `{ "characters": [] }`
* `work_context.json` → `{ "description": "" }`
* `term_memory.json` → `{}`
*(اختياري: استخراج بعض الأسماء الأولية من النص).*

---

## 7) التقطيع (Chunking)
* **حجم الشنك:** 20 سطر.
* **السياق (Context):** 
  * 5 أسطر سابقة (من الشنك المترجم السابق).
  * سطرين لاحقين (من الشنك القادم الأصلي).

---

## 8) طبقة الذكاء الاصطناعي (LLM Layer)
يجب أن يكون الـ Wrapper الخاص بالاستدعاء صلباً ليدعم:
* **Retry:** إعادة المحاولة حتى 5 مرات.
* **Exponential Backoff:** مضاعفة وقت الانتظار عند الفشل لتجنب الـ Rate limit.
* **Timeout:** مهلة قصوى للطلب.
* **Fallback:** آليات بديلة.
* **في حال الفشل النهائي:** 
  → وسم الشنك بـ `degraded`.
  → استخدام النص الإنجليزي الأصلي لتجاوز التوقف.

---

## 9) مستخرج البيانات (JSON Extractor)
مهامه:
* تنظيف أي نصوص أو تعليقات إضافية (Markdown/Text) يولدها النموذج حول الـ JSON.
* استخراج المصفوفة بشكل نظيف `[ {...}, {...} ]`.

---

## 10) المدقق (Validator)
التحقق الصارم من كل مخرج:
* جميع الأرقام المرجعية (IDs) موجودة ومطابقة.
* عدم وجود حقول ترجمة فارغة.
* بنية JSON صحيحة.
* احترام القاموس (Glossary respected) بشكل جزئي.
* **عند الفشل:** → `Retry`.

---

## 11) محرك القيود (Constraint Engine)
تطبيق القيود البصرية للـ Subtitle:
* **الحد الأقصى للسطر:** (Max line length).
* **التقسيم الآلي:** (Split) لسطرين كحد أقصى.
* **الضغط الآلي:** (Compression) لتقليل الأحرف بدون المساس بالمعنى.
* **مؤشر CPS:** (Characters Per Second) ويحسب: `CPS = length / duration`.
* **الأوضاع (Modes):** `Strict`, `Balanced`, `Off`.

---

## 12) ذاكرة المصطلحات (Term Memory)
تتبع المصطلحات تلقائياً وتثبيتها.
* **الهيكل المطلوب:**
```json
{
  "master": {
    "translation": "سيدي",
    "count": 12,
    "locked": true
  }
}
```
* **القواعد:**
  * التحديث بعد كل شنك مترجم.
  * لا يحدث أثناء التوازي (لمنع الـ Race conditions).
  * الحد الأقصى للمصطلحات `max = 50`.
  * **الإخلاء (Eviction):** يحذف الأقل استخداماً ثم الأقدم.
  * **القفل:** إذا كان `locked: true` → لا يمسح ولا يتغير.

---

## 13) نظام السجلات (Logging)
* **لكل شنك (Per Chunk):**
```json
{
  "chunk_id": 1,
  "status": "success",
  "retries": 1,
  "duration": 2000
}
```
* **المقاييس الشاملة (Metrics):**
  * نسبة إعادة المحاولة (Retry rate).
  * نسبة الفشل (Failure rate).
  * نسبة تحسين القراءة (CPS improvement).

---

## 14) واجهة المستخدم (GUI)
مقسمة لثلاثة تبويبات (Tabs):
1. **Run:** التشغيل والمتابعة.
2. **Settings:** الإعدادات العامة.
3. **Data Editor:** إدارة قواعد البيانات.

---

## 15) تبويب التشغيل (Run Tab)
* زر اختيار ملف الترجمة.
* عرض تفاصيل الأنمي ورقم الحلقة المكتشف.
* أزرار التحكم: (Start / Resume / Stop).
* شريط تقدم (Progress Bar).
* نافذة لعرض السجلات (Logs terminal).

---

## 16) تبويب الإعدادات (Settings Tab)
* **Model:** قائمة منسدلة للمحركات (OpenAI, DeepSeek, etc).
* **API Key:** حقل الإدخال مع زر (Test connection).
* **Paths:** تحديد مسارات ملفات القاموس، الشخصيات، وسياق العمل.
* **Output:** مجلد الحفظ الافتراضي.

---

## 17) تبويب محرر البيانات (Data Editor Tab)
تتيح واجهة للإضافة، الحذف، التعديل، والـ (Lock).
* **(A) Characters:** جدول `| Name | Gender |`
* **(B) Work Context:** حقل نصي كبير `Textarea`.
* **(C) Glossary:** جدول `| Term | Translation | Type |`
* **(D) Term Memory:** جدول `| Term | Translation | Count | Locked |`

---

## 18) الربط بين الواجهة والمشروع
* **عند اختيار ملف:** الواجهة تكتشف الأنمي (`detect anime`) وتعيّنه كـ `current_project`.
* **محرر البيانات (Data Editor):** يقرأ محتواه ويحفظه تلقائياً من المسار الدقيق `project/data`.

---

## 19) المشغل (Runner)
* يعتمد على مكتبة `subprocess` لتشغيل سكربت بايثون.
* **القاعدة:** لا يحجب واجهة المستخدم (No blocking).
* يستمع للسجلات (Logs) ويقوم بتحديث الـ Progress bar في الـ GUI.

---

## 20) الإعدادات العامة (Config)
يُخزن الـ Config بصيغة JSON:
```json
{
  "model": {},
  "paths": {},
  "output": {},
  "execution": {}
}
```

---

## 21) التسامح مع الأخطاء (Fault Tolerance)
النظام يجب أن لا ينهار:
* دعم لفشل الشبكة المؤقت (Network failure).
* الـ Retries التلقائي.
* آليات التراجع (Fallback).
* مفتاح الأمان (Circuit breaker) في حالة استنفاد المحاولات.

---

## 22) أنماط التنفيذ (Execution Modes)
* **الآن (Currently):** ✔ متسلسل فقط (Sequential) لضمان استقرار ذاكرة المصطلحات.
* **لاحقاً (Later):** ✔ متوازي (Parallel) مع تطوير آلية لفصل إدارة הـ Term memory بعيداً عن التداخل.

---

## 23) قواعد ذهبية حاسمة (أوامر التنفيذ)
* ❌ **لا تربط الـ GUI بالكود الأساسي (Core) بشكل مباشر:** استخدم الـ Runner و الـ Subprocess.
* ❌ **لا تعتمد على الـ LLM في كل شيء:** دع المهام المنطقية والتنسيقية للبايثون.
* ❌ **لا تكسر هيكل JSON:** تأكد من الـ Validators دوماً.
* ❌ **لا تجعل المستخدم يكتب ملفات JSON يدوياً:** وفر الواجهة (GUI) لذلك دائماً.

---

## 24) قواعد تجربة المستخدم (UX)
* ✔ الاكتشاف والبناء الآلي لمشروع الأنمي وحلقته.
* ✔ التقليل من الأسئلة أو الخيارات المزعجة للمستخدم.
* ✔ السماح بتعديل كل البيانات اللاحقة (Glossary / Terms / Settings).
